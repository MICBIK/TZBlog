### 目标 2: 补充专项审计

#### 2.1 并发安全审计

**审计范围**:
- `internal/cache/` - 缓存并发访问
- `internal/service/` - 业务逻辑并发安全
- `internal/repository/` - 数据库并发操作
- `internal/api/middleware/` - 中间件并发安全

**审计清单**:
- [ ] 数据竞争检测（`go run -race`）
- [ ] Goroutine 泄漏检测
- [ ] Channel 使用正确性
- [ ] Mutex 使用正确性
- [ ] 原子操作正确性
- [ ] Context 传递正确性
- [ ] 超时控制完整性
- [ ] 资源清理完整性

**工具**:
```bash
# 数据竞争检测
go test -race ./...

# Goroutine 泄漏检测
go test -count=100 ./internal/service/...

# 静态分析
go vet ./...
staticcheck ./...
```

#### 2.2 API 设计审计

**审计范围**:
- `internal/api/handlers/` - 所有 API handler
- `internal/api/middleware/` - 中间件
- `pkg/response/` - 响应格式

**审计清单**:
- [ ] RESTful 规范遵守
- [ ] HTTP 方法正确性
- [ ] 状态码正确性
- [ ] 响应格式统一性
- [ ] 错误信息清晰度
- [ ] 分页支持完整性
- [ ] 过滤排序支持
- [ ] API 版本控制
- [ ] Swagger 文档完整性
- [ ] 向后兼容性

**检查点**:
1. **路由设计**
   - GET 用于读取
   - POST 用于创建
   - PUT 用于完整更新
   - PATCH 用于部分更新
   - DELETE 用于删除

2. **状态码使用**
   - 200 OK - 成功
   - 201 Created - 创建成功
   - 204 No Content - 删除成功
   - 400 Bad Request - 请求错误
   - 401 Unauthorized - 未认证
   - 403 Forbidden - 无权限
   - 404 Not Found - 资源不存在
   - 500 Internal Server Error - 服务器错误

3. **响应格式**
   ```go
   type Response struct {
       Success bool        `json:"success"`
       Data    interface{} `json:"data,omitempty"`
       Error   *Error      `json:"error,omitempty"`
       Meta    *Meta       `json:"meta,omitempty"`
   }
   ```

### 目标 3: 修复剩余 24 个 HIGH 级问题

#### 3.1 问题分类

根据 Phase 3 审计报告，24 个 HIGH 问题分类如下：

| 类型 | 数量 | 优先级 |
|------|------|--------|
| **安全相关** | ~6 | P0 |
| **性能优化** | ~6 | P1 |
| **代码质量** | ~5 | P2 |
| **数据库设计** | ~4 | P2 |
| **API 设计** | ~3 | P3 |

#### 3.2 修复策略

**Phase 1: 安全相关（P0）**
预计时间: 2-3 小时

1. 输入验证不完整
   - 补充缺失的输入验证
   - 添加长度限制
   - 添加格式验证

2. 权限检查遗漏
   - 补充资源所有权检查
   - 添加细粒度权限控制

3. 敏感信息日志记录
   - 移除日志中的敏感信息
   - 添加日志脱敏

4. 错误信息泄露
   - 统一错误信息格式
   - 移除内部实现细节

5. Session 管理问题
   - 添加 session 超时
   - 添加 session 撤销机制

6. API 限流缺失
   - 添加全局限流
   - 添加端点级限流

**Phase 2: 性能优化（P1）**
预计时间: 2-3 小时

1. N+1 查询问题
   - 添加预加载（Preload）
   - 使用 JOIN 优化

2. 缺失的数据库索引
   - 分析慢查询
   - 添加必要索引

3. 低效的查询
   - 优化全表扫描
   - 添加查询条件

4. 缓存缺失
   - 添加热数据缓存
   - 设置合理过期时间

5. 缓存过期策略
   - 统一缓存 TTL
   - 添加缓存失效机制

6. 批量操作优化
   - 使用批量插入
   - 使用事务优化

**Phase 3: 代码质量（P2）**
预计时间: 1-2 小时

1. 函数过长（>50 行）
   - 拆分大函数
   - 提取辅助函数

2. 圈复杂度过高
   - 简化条件逻辑
   - 使用策略模式

3. 错误处理不一致
   - 统一错误处理
   - 使用标准错误类型

4. 命名不规范
   - 重命名不清晰的变量
   - 遵循 Go 命名规范

5. 重复代码
   - 提取公共逻辑
   - 创建工具函数

**Phase 4: 数据库设计（P2）**
预计时间: 1-2 小时

1. 缺失外键约束
2. 缺失唯一索引
3. 字段类型不合理
4. 缺失默认值

**Phase 5: API 设计（P3）**
预计时间: 1 小时

1. RESTful 规范偏差
2. 响应格式不统一
3. 缺失分页支持
