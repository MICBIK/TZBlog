## 🔴 BLOCKER 级别问题（3个）

### B1. 缺失 Service 层导致业务逻辑泄漏

**位置**: `internal/api/handlers/`  
**影响**: 架构健壮性

Handler 包含大量业务逻辑（95行+），职责过重，无法复用。

**建议**: 引入 Service 层封装业务逻辑

### B2. 缺少统一的错误类型体系

**位置**: 全局  
**影响**: 错误处理混乱

数据库错误、业务错误无法区分，错误信息硬编码。

**建议**: 定义统一的 `AppError` 类型

### B3. CORS 配置存在安全风险

**位置**: `internal/api/middleware/cors.go:10-11`  
**影响**: 安全漏洞

```go
Access-Control-Allow-Origin: *
Access-Control-Allow-Credentials: true
```

这是违反 CORS 规范的危险配置。

**建议**: 使用白名单验证 Origin

## 🔴 CRITICAL 级别问题（22个）

### 安全漏洞（10个）

#### SEC-001: JWT 算法混淆攻击 ⚠️⚠️⚠️

**CWE**: CWE-347  
**位置**: `pkg/auth/jwt.go:39`  
**严重性**: CRITICAL

**问题**: `jwt.ParseWithClaims` 未验证签名算法，攻击者可构造使用 `none` 算法的 token，完全绕过身份验证，伪造 admin 身份。

**修复**:
```go
token, err := jwt.ParseWithClaims(tokenString, &Claims{}, func(token *jwt.Token) (interface{}, error) {
    if _, ok := token.Method.(*jwt.SigningMethodHMAC); !ok {
        return nil, fmt.Errorf("unexpected signing method: %v", token.Header["alg"])
    }
    return []byte(secret), nil
})
```

#### SEC-002: Token 撤销机制缺失

**CWE**: CWE-613  
**严重性**: CRITICAL

用户修改密码、注销后，JWT token 仍可使用 7 天。

**修复**: 实现 Redis 黑名单或 Token 版本号机制

#### SEC-003: 生产环境弱 JWT Secret

**CWE**: CWE-321  
**严重性**: CRITICAL

示例配置使用弱密钥，如果生产未更改，可被破解。

**修复**: 启动时强制验证 Secret 强度（>32字符）

#### SEC-004: 缺少登录失败限流

**CWE**: CWE-307  
**严重性**: HIGH

Login 端点无限流，可被暴力破解。

**修复**: 添加基于 IP 和账户的限流

#### SEC-005: CORS 配置致命漏洞

**严重性**: CRITICAL

允许任意来源+携带凭据，攻击者可窃取 token。

#### SEC-006: 缺少 CSRF 防护

**CWE**: CWE-352  
**严重性**: CRITICAL

所有状态变更接口都缺少 CSRF 防护。

#### SEC-007: XSS - 文章和评论未转义

**CWE**: CWE-79  
**严重性**: HIGH

文章内容和评论未转义，可注入恶意脚本。

#### SEC-008: 文件上传仅验证扩展名

**严重性**: HIGH  
**位置**: `pkg/storage/r2.go:79-81`

未验证 MIME 类型，可上传伪装的恶意文件。

#### SEC-009: 整数转换错误未处理

**严重性**: HIGH

`strconv.Atoi` 错误被忽略，可能导致逻辑错误。

#### SEC-010: JWT 过期时间过长

**严重性**: HIGH

Access token 有效期 7 天，过长。

**建议**: 缩短至 15-60 分钟，使用 refresh token

### 数据库问题（5个）

#### DB-001: 时间戳类型不一致 ⚠️

**严重性**: CRITICAL

- SQL: `TIMESTAMP`
- Go: `int64` (Unix 时间戳)
- Go: `time.Time`

三种表示方式共存，导致 ORM 映射错误。

**修复**: 统一使用 `time.Time`

#### DB-002: 缺少数据完整性约束

**严重性**: CRITICAL

`likes` 表 `target_id` 无外键约束，可能引用不存在的记录。

**修复**: 拆分为 `article_likes` 和 `comment_likes`，添加外键

#### DB-003: orders 表金额精度不足

**严重性**: CRITICAL

`DECIMAL(10,2)` 最大只能到 99,999,999.99

**修复**: 改为 `DECIMAL(12,2)` 或 `DECIMAL(15,2)`

#### DB-004: 软删除字段类型不一致

**严重性**: HIGH

SQL 定义 `TIMESTAMP`，但代码写入 Unix 时间戳（整数）。

#### DB-005: 缺少关键索引

**严重性**: HIGH

缺少 5+ 个关键复合索引，导致慢查询。

### 性能问题（3个）

#### PERF-001: N+1 查询问题

**严重性**: HIGH  
**位置**: `repository/postgres/article_repo.go:76-77`

加载 20 篇文章 = 41 次查询（1 + 20 + 20）

**预计影响**: ~10ms × 41 = 410ms

**修复**: 使用 JOIN 或优化 Preload

#### PERF-002: 重复 COUNT 查询

**严重性**: HIGH  
**位置**: `stats_repo.go:18-40`

6 次独立 COUNT 查询可合并为 1 次。

#### PERF-003: 缓存穿透防护缺失

**严重性**: HIGH

无缓存穿透、雪崩防护机制。

### 并发问题（2个）

#### CONC-001: Goroutine 泄漏

**严重性**: CRITICAL  
**位置**: `middleware/ratelimit.go:55-62`

每次调用都启动新 goroutine，无停止机制。

#### CONC-002: Context 超时缺失

**严重性**: HIGH

Redis/R2 操作缺少超时 context，可能无限阻塞。

### 代码质量问题（2个）

#### CODE-001: 核心字段缺少验证

**严重性**: CRITICAL

Article.Title、User.Email 等核心字段无验证逻辑。

#### CODE-002: Payment Status 使用魔法字符串

**严重性**: HIGH

应使用类型化枚举。
