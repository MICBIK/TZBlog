## 🔴 BLOCKER 级别问题

### B1. 缺失 Service 层导致业务逻辑泄漏

**发现者**: architecture-1  
**位置**: `internal/api/handlers/article_handler.go:34-84`

**问题描述**:
Handler 包含 95 行复杂业务逻辑（生成 slug、计算阅读时长、发布判断等），职责过重，无法复用。

**影响**:
- 业务逻辑无法在 CLI/定时任务中复用
- 测试困难，必须模拟 HTTP 上下文
- 事务管理缺失

**修复方案**: 引入 Service 层封装业务逻辑

**优先级**: P0 - 架构重构基础

---

### B2. 缺少统一的错误类型体系

**发现者**: architecture-1  
**位置**: 全局所有 Handler

**问题描述**:
错误处理混乱，数据库错误、业务错误无法区分，错误信息硬编码。

**影响**:
- Handler 无法区分"未找到"和"数据库故障"
- 无法国际化
- 缺少错误码追踪

**修复方案**: 定义统一的 `AppError` 类型

**优先级**: P0 - 影响所有 API

---

### B3. CORS 配置存在安全风险

**发现者**: architecture-1  
**位置**: `internal/api/middleware/cors.go:10-11`

**问题描述**:
```go
c.Writer.Header().Set("Access-Control-Allow-Origin", "*")
c.Writer.Header().Set("Access-Control-Allow-Credentials", "true")
```

`Allow-Origin: *` + `Allow-Credentials: true` 违反 CORS 规范，浏览器会拒绝。

**修复方案**: 使用白名单验证 Origin

**优先级**: P0 - 安全漏洞

---

#### C13.2: 全局可变状态 - 配置单例

**发现者**: performance-2  
**位置**: `config/config.go:78`

**问题**:
```go
var cfg *Config  // 全局变量

func Load() (*Config, error) {
    cfg = &config  // race condition
    return cfg, nil
}
```

**风险**: 测试时 race condition、配置重载时部分读取

**修复**: 移除全局变量，通过依赖注入传递

**优先级**: P0 - 立即修复

---

#### DB.1: 时间戳类型不一致

**发现者**: database-1  
**位置**: migrations/*.sql vs domain/*.go

**问题**:
- SQL: `TIMESTAMP`
- Go: `int64` (Unix 时间戳)

**风险**: ORM 映射错误

**修复**: 统一使用 `time.Time` 或全部改为 `BIGINT`

**优先级**: P0 - 阻塞性问题

---

#### DB.2: 缺少数据完整性约束

**发现者**: database-1

**问题**:
- `likes` 表 `target_id` 无外键约束
- `articles.author_id` CASCADE 删除但 `category_id` SET NULL，策略不一致

**修复**: 拆分 `likes` 表为 `article_likes` 和 `comment_likes`，添加外键

**优先级**: P0

---

#### DB.3: orders 表金额精度不足

**发现者**: database-1  
**位置**: migrations

**问题**: `amount DECIMAL(10,2)` 最大只能到 99,999,999.99

**修复**: 改为 `DECIMAL(12,2)` 或 `DECIMAL(15,2)`

**优先级**: P0 - 财务数据
