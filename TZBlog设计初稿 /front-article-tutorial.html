我用 spec-first 工作流让 Claude 连续写对 3000 行代码

HAhaiden ·2026-05-28 ·22 min ·3.2k 阅读
AI Codingpromptagent工作流
spec → verify → replay · 一个能回放的失败轨迹

过去半年我让 agent 写过最长的一次，是一个 3000 行的支付对账模块——一次性跑通、过了全部契约测试、上了生产。很多人以为关键是"更聪明的模型"或"更长的 prompt"。都不是。真正起作用的，是我把功能开工的顺序倒过来了：先写验证，再写实现，让 agent 的唯一目标变成"让验证全绿"。

为什么 prompt 越写越长反而越糟

越长的 prompt 越像"许愿"——你在描述一个你脑子里的成品，但 agent 没有办法判断它有没有命中。它会很自信地写出一坨看起来对、跑起来错的代码。问题不在它的能力，在于没有一个客观的、它够得到的成功信号。

  • 需求在你脑子里是模糊的，写成 prompt 时被迫具体化，但具体得不够
  • agent 把"看起来合理"当成"正确"，因为没有更硬的标准
  • 你在 review 时才发现偏差，返工成本被推到了最后

spec-first：先写验收脚本

我现在每个功能开工前先落的，不是 prompt，而是一段验收脚本。它先于实现存在，agent 的任务就是让它从红变绿：

bashverify.sh
1234567
# 验收脚本先于实现存在——agent 的唯一目标是让它全绿
set -euo pipefail

pnpm tsc --noEmit     # 类型先过
pnpm vitest run spec/checkout  # 行为契约
node scripts/replay.js "last-failure"  # 回放上次失败轨迹
echo "✓ spec satisfied"

这三行分别对应三个闸门：类型挡掉一半的低级错误，契约测试把"我以为它懂了"变成"它必须通过"，失败回放让上次踩过的坑不会重复掉进去。

核心洞察
关键不是脚本写得多漂亮，而是它在 agent 动手之前就存在。验证层是 agent 的"目标函数"——没有它，再聪明的模型也只是在猜你想要什么。

最便宜的验证层放在哪

不是所有验证都值得写。我的排序是：类型 > 纯函数单测 > 契约测试 > e2e。越靠左越便宜、反馈越快，越应该让 agent 频繁撞它。e2e 留到最后做兜底，而不是开发循环里的主力。

一个反直觉的点

很多人省略类型检查，觉得"反正测试会兜住"。但类型是 agent 写代码时就能拿到的反馈，测试要跑起来才知道。把最快的反馈给 agent，它的收敛速度会肉眼可见地提升。

失败轨迹回放

这是最被低估的一环。每次 agent 失败，我把它的输入、输出、报错存成一条 trace。下次开工先 replay 一遍历史失败——等于给 agent 装了"长期记忆"，同一个坑不会掉第二次。半年下来，这个 trace 库本身成了团队最值钱的资产。
