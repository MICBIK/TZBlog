# TZBlog 全新设计方案

## 1. 方案定位

TZBlog 不再做“纯 HTML/CSS/JS 的宇宙主题单页展示站”，而是做一套适合长期写作、项目沉淀、文档归档和个人表达的内容型博客系统。

保留你原方案里的宇宙主题、轨道意象、SVG 表达和沉浸式氛围，但整体方向改成：

- 内容优先
- 页面清楚
- 导航稳定
- 动效克制
- 可持续扩展

一句话定义：

`一个以深空观测站为视觉母题、以内容检索和长期写作为核心的技术博客系统。`

## 2. 现有方案的问题

当前这版早期宇宙单页方案更像一个“交互创意提案”，不是可直接开发的博客产品方案，主要问题有：

- 过于偏单页作品集，不适合文章、项目、文档长期增长
- 交互点很多，但信息架构不够清楚
- 拖拽、粒子、星座绘制这类效果很酷，但和博客主任务关联弱
- 纯原生单页方案后期维护成本高，内容检索和模板复用能力弱
- 没有把“首页、列表页、文章页、项目页、搜索页”拆成清晰的产品结构

所以这次重做不是润色原稿，而是重新建立产品骨架。

## 3. Firefly 可借鉴的核心思路

参考 Firefly 后，可以明确借鉴以下方向。

### 3.1 架构层

- 采用内容驱动而不是页面硬编码
- 采用配置驱动而不是把所有布局逻辑写死在页面里
- 用布局系统统一处理首页、列表页、文章页的主框架
- 搜索、目录、侧栏、主题切换、展示控制都做成独立模块

Firefly 最值得借鉴的，不是“长得像它”，而是：

`用 config、layout、feature、content 四层结构，把博客做成可组合系统。`

### 3.2 界面层

- 首页不是纯展示横幅，而是“横幅 + 主内容栅格”的组合
- 侧栏不是装饰，而是承载资料、导航、分类、标签、统计、目录等辅助信息
- 列表页支持不同信息密度，适合内容增长后继续使用
- 页面有统一的卡片语言、统一的层级语言、统一的视觉节奏

### 3.3 动效层

- 页面切换动效很轻，不抢内容
- 首页标题、背景、波浪、浮动控件都服务于氛围和导航
- 动效是“可关闭 / 可切换 / 可降级”的，不是强绑定

### 3.4 不直接照搬的部分

Firefly 里有些东西适合主题模板，但不一定适合 TZBlog 主定位：

- Live2D
- 音乐播放器
- 樱花特效
- 过于二次元的横幅表达
- 过多的用户可调控制面板

TZBlog 应该更偏“科技感、轨道感、观测站感”，而不是二次元装饰站。

### 3.5 Coff0xc 可借鉴的部分

- Hero 到正文列表的过渡非常自然，内容入口明确
- 文章页、目录、搜索、标签、分页这些博客基本能力很完整
- 统一卡片语言让首页、列表页、详情页保持一致
- 技术博客感强，但不显杂乱

对 TZBlog 的启发是：

`宇宙主题不能削弱检索和阅读能力，内容页必须像成熟博客，而不是展示页附属品。`

### 3.6 Anna 可借鉴的部分

- 深空背景与前景内容层分离得很干净
- Three.js 主行星的存在感很强，但没有压住正文
- 行星效果依靠光照、纹理、环带和缓慢镜头反馈建立氛围
- 整体是“主视觉很强，内容界面很克制”的结构

对 TZBlog 的启发是：

`把宇宙感集中在首页 Hero 和少量横幅区，而不是把整个博客做成满屏乱飞的特效场。`

## 4. TZBlog 的新设计方向

### 4.1 核心概念

新的视觉母题定为：

`Orbital Observatory / 轨道观测站`

它不是把页面真的做成一堆会拖来拖去的星球，而是把宇宙元素转译成更容易阅读的界面语言：

- 星图 = 全站导航结构
- 轨道 = 内容流转关系
- 舱室 / 面板 = 各类卡片模块
- 信号线 = 页面中的分割与引导
- 星体发光 = 重点内容强调
- 航迹 = 页面切换与滚动反馈

### 4.2 产品目标

- 让用户一进首页就知道站点主要内容在哪里
- 让文章、项目、文档三类内容有明确入口，不混杂
- 让宇宙主题成为识别度，而不是阅读负担
- 让后续新增内容时不需要推翻结构

## 5. 信息架构

### 5.1 一级导航

建议固定为以下结构：

- 首页
- 文章
- 项目
- 文档
- 实验室
- 关于
- 搜索

说明：

- `首页` 承担世界观、身份表达和内容分发
- `文章` 放博客正文、随笔、技术分享、复盘
- `项目` 放完整项目介绍、进展、截图、技术栈、里程碑
- `文档` 放教程、方案、工具说明、开发记录
- `实验室` 放小工具、Demo、概念验证、交互实验
- `关于` 放个人介绍、技术栈、时间线、联系方式
- `搜索` 不做隐藏功能，要作为主入口存在

### 5.2 内容类型

建议从一开始就拆成 5 类内容模型：

- `posts`
  正常博客文章
- `projects`
  项目详情页
- `docs`
  长期维护的说明文档
- `notes`
  短笔记 / 开发碎片 / 更新记录
- `pages`
  关于页、友链页、站点说明等静态页

### 5.3 首页结构

首页不再是“单屏宇宙秀场”，而是分成 6 个清晰区块。

1. 首屏 Hero
   站点标题、身份描述、当前状态、主 CTA、轨道型导航入口
2. Focus Stream
   最新文章、重点项目、持续更新文档
3. Mission Panels
   文章 / 项目 / 文档 / 实验室四个主入口
4. Selected Works
   精选项目或系列专题
5. Timeline / Changelog
   最近更新、开发日志、站点变化
6. Footer Dock
   联系方式、订阅、站点说明、版权与技术栈

### 5.4 文章页结构

文章页建议采用“阅读主轴 + 辅助面板”的布局。

- 顶部：标题、摘要、发布日期、标签、阅读时长
- 主体：正文内容，保持舒适阅读宽度
- 右侧：目录、系列导航、相关文章
- 左侧或顶部：作者信息、文章状态、上一篇 / 下一篇
- 底部：引用来源、评论区、推荐阅读

### 5.5 项目页结构

项目页不能只是一篇普通文章，需要更像“项目档案”。

- 项目概述
- 当前状态
- 技术栈
- 关键截图
- 核心功能
- 架构说明
- 开发日志
- 在线地址 / 仓库 / 文档链接

### 5.6 文档页结构

文档页应更强调检索和结构化阅读。

- 文档目录
- 当前版本 / 更新时间
- 章节导航
- 相关文档推荐
- 回到项目或文章的关联入口

## 6. 布局系统

### 6.1 总体策略

借鉴 Firefly 的布局思路，但做更适合 TZBlog 的三套固定模板：

- `Home Layout`
- `Content Index Layout`
- `Article / Doc Detail Layout`

不建议一开始就开放太多布局切换项，而是先把默认体验打磨成熟。

### 6.2 首页布局

桌面端建议采用：

- 顶部横幅 + 下沉主面板
- 左侧身份区
- 中部内容主流
- 右侧快捷模块

这能兼顾宇宙氛围和信息密度，也符合技术博客首页的使用习惯。

### 6.3 列表页布局

`文章`、`项目`、`文档` 三类列表页建议统一支持两种视图：

- 列表视图
  适合深度阅读和大摘要
- 卡片网格视图
  适合快速浏览和高密度扫描

但默认策略建议是：

- 文章默认列表
- 项目默认网格
- 文档默认列表

### 6.4 详情页布局

详情页建议做成“中间主阅读区 + 右侧目录 / 相关模块”的稳定结构。

如果桌面宽度足够，可以在左侧增加一个窄栏，展示：

- 作者信息
- 标签
- 状态
- 当前系列

移动端则全部折叠为单列，目录改成浮动按钮或底部抽屉。

## 7. 视觉系统

### 7.1 整体气质

视觉关键词：

- 深空
- 冷静
- 精密
- 观测屏
- 航迹
- 信号感

不是卡通宇宙，也不是纯黑客终端，而是：

`带有空间观测仪器感的现代技术博客。`

### 7.2 色彩方案

建议使用以下主色体系：

- 背景主色：`#08111f`
- 背景次色：`#0d1728`
- 面板色：`rgba(11, 23, 40, 0.72)`
- 主强调色：`#67e8f9`
- 次强调色：`#8b9cff`
- 警示 / 亮点色：`#f59e0b`
- 文本主色：`#e6eef8`
- 文本次色：`#95a6bf`
- 边框色：`rgba(103, 232, 249, 0.18)`

### 7.3 字体建议

- 中文正文：`Noto Sans SC`
- 英文标题：`Space Grotesk`
- 代码与数据：`JetBrains Mono`

这样可以同时保证中文可读性、英文标题识别度和技术气质。

### 7.4 图形语言

图形元素不再使用大量独立星球堆满页面，而改成：

- 轨道线
- 星图网格
- 雷达环
- 微弱粒子层
- 舱室式卡片边框
- 发光角标与扫描线

宇宙元素主要集中在首页 Hero、分页横幅和少量插图区域，不要污染正文阅读。

### 7.5 首页主视觉修订

在现有方案基础上，首页主视觉明确改为：

- 一个具备土星气质的主行星
- 少量卫星和轨道线
- 深空背景、星图层、弱光晕层
- 标题和导航压在前景内容层

也就是说，首页不再做“很多星球同时抢注意力”的结构，而是做：

`一个明确主星体 + 清晰内容入口` 的首屏。

## 8. 动效系统

### 8.1 动效原则

- 只对导航、状态反馈和氛围层做动效
- 不让动效打断阅读
- 所有动效都要能降级
- 手机端动效密度要明显低于桌面端

### 8.2 推荐动效

1. 页面切换
   使用轻量级上下浮动 + 透明度过渡，时长控制在 140ms 到 220ms
2. Hero 背景
   缓慢轨道旋转、星尘漂移、弱呼吸光效
3. 卡片悬浮
   微抬升、边框高亮、阴影变亮
4. 导航反馈
   激活项出现“航迹线”或扫描条
5. 文章目录定位
   当前章节有高亮脉冲或细线进度反馈
6. 首屏主标题
   可保留打字机或逐字显现，但只用于首屏一句话，不要泛滥

### 8.3 不建议保留的旧交互

以下内容建议不作为第一版核心功能：

- 全页面点击粒子爆炸
- 随处拖拽生成彗星尾迹
- Shift + 拖拽绘制星座
- 大量可拖动行星组件

这些更像互动展示页特性，不适合博客的长期使用场景。

### 8.4 可作为彩蛋保留的交互

如果后续你仍想保留一点“宇宙趣味”，建议只保留在首页：

- Hero 区星图轻微视差
- Logo 悬浮时的轨道亮起
- 首页主入口卡片 hover 时出现短暂扫描动画
- 主行星随鼠标产生极轻微镜头偏移

## 9. 技术架构建议

### 9.1 推荐技术栈

建议采用：

- 前台：Astro
- 后台：Payload CMS
- 数据库：PostgreSQL
- 统计：Umami
- 搜索：Pagefind
- 媒体：S3 / R2

局部交互组件可按需引入少量 Islands，首页 Hero 的 3D 层由 Three.js 承担，但前台仍然保持 Astro 主导。

### 9.2 选择这套栈的原因

- Astro 继续承担高质量内容前台，保留页面和视觉自由度
- Payload 提供完整后台，不需要自己重造文章发布、媒体管理、草稿版本和权限系统
- PostgreSQL 适合作为主内容库
- Umami 直接提供现成的数据统计与事件分析面板
- Pagefind 继续适配 Astro 前台搜索
- 这套组合比直接套 Ghost / Halo 更适合你这种“自定义前台 + 多内容模型”的博客

### 9.3 建议目录结构

```text
TZBlog/
├── apps/
│   ├── web/
│   │   ├── public/
│   │   └── src/
│   └── cms/
│       ├── src/
│       └── payload.config.ts
├── packages/
│   ├── shared-types/
│   └── design-tokens/
├── infra/
│   ├── docker-compose.yml
│   └── env/
└── docs/
```

### 9.4 配置驱动原则

从第一版就把以下内容抽到配置层：

- 站点基本信息
- 顶部导航
- 首页模块顺序
- 社交链接
- 主题色和视觉变量
- 文章 / 项目 / 文档列表展示模式

同时，以下内容交给后台管理：

- 导航
- 首页模块文案
- 社交链接
- SEO 默认配置
- 文章 / 项目 / 文档内容本体

这样前台负责表现，后台负责内容，不会陷入全站硬改。

### 9.5 后台与统计策略

- Payload 负责内容管理、草稿、版本、媒体、权限
- Umami 负责 PV、UV、来源、事件统计
- Astro 负责公开访问页面和 SEO
- 内容发布通过 Webhook 触发 Astro 重新构建

## 10. 当前设计与原 Cosmic 方案的融合方式

保留的部分：

- 宇宙主题
- 轨道意象
- SVG / 图形化表达
- 首屏沉浸氛围
- 文章、项目、文档并存的多内容结构

重构的部分：

- 单页改多页
- 炫技交互改产品交互
- 随机视觉元素改统一设计语言
- 纯前端脚本式结构改配置驱动内容架构

删除或降级的部分：

- 高噪声鼠标特效
- 影响阅读的强动效
- 与内容无关的拖拽玩法

## 11. 开发阶段建议

### Phase 1：骨架

- 初始化 Astro 前台
- 初始化 Payload 后台
- 建立 PostgreSQL 与媒体存储
- 建立基础内容模型与布局骨架

### Phase 2：视觉

- 建立设计变量
- 完成首页 Hero 与主面板
- 完成卡片系统、标签系统、目录系统
- 完成响应式布局

### Phase 3：功能

- 搜索
- 标签 / 分类 / 系列页
- 相关文章
- 项目状态与更新时间系统
- Payload 发布链路
- Umami 统计与埋点

### Phase 4：动效与润色

- 页面切换
- Hero 轨道动画
- Hover 反馈
- 滚动进入动画
- reduced-motion 支持

## 12. 第一版验收标准

第一版不以“特效多”为标准，而以以下结果为准：

- 首页一眼能看懂站点结构
- 文章、项目、文档入口清楚
- 文章页阅读体验稳定
- 搜索和列表浏览效率高
- 宇宙主题明确但不影响信息获取
- 后续写新内容时不需要重做架构

## 13. 最终结论

TZBlog 的正确方向不是“把宇宙效果堆得更满”，而是：

`借用 Firefly 的系统化博客思路，保留 Astro 前台设计自由度，再用 Payload 和 Umami 补齐完整后台与统计能力，做成一套能长期维护的完整前后端宇宙博客。`

如果后续进入开发阶段，这份文档就应作为首页结构、页面模板、样式变量、动画边界和目录结构的统一基线。
