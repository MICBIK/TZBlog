## 🎓 经验教训

### 1. 审计报告不可全信

**问题**: 原报告 22 个 BLOCKER，实际只有 6 个

**教训**: 
- 修复前必须逐一验证代码
- 优先处理可验证的基础设施问题
- 对"集成问题"类断言保持怀疑

### 2. 自动化工具有局限

**问题**: 静态分析产生大量误报

**教训**:
- 工具扫不到运行时行为（拦截器解包）
- 工具不懂业务逻辑（列表不返 Author）
- 需人工复核关键发现

### 3. 多轮审计需同步

**问题**: README 和 SUMMARY 矛盾

**教训**:
- 修复后更新所有相关报告
- 使用单一事实来源（代码）
- 定期重新验证

### 4. 优先修复可证伪的问题

**最可靠的问题**:
- ✅ 构建/测试失败（可运行验证）
- ✅ 文件缺失（可 find 验证）
- ✅ 代码覆盖率（可工具测量）

**需谨慎的问题**:
- ⚠️ "未实现某功能"（可能换了名字）
- ⚠️ "性能问题"（需实测，不能靠推断）
- ⚠️ "安全隐患"（需上下文，不能只看片段）
